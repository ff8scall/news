## 🛠️ Prerequisites: Tools & Environment Setup

Before we begin, please ensure your environment meets these strict requirements.

Success depends on precise version matching.

*   **OS:** Linux (Ubuntu 22.04 recommended).

*   **Python:** 3.10 or 3.11 (3.12+ compatibility issues are common).

*   **GPU:** NVIDIA GPU (Minimum 12GB VRAM recommended for fine-tuning).

*   **Drivers:** Latest CUDA Toolkit (12.1 or newer) and cuDNN library.

*   **Core Libraries:** PyTorch (2.2+) and Hugging Face Transformers.

> **System Check:** Verify your GPU drivers and CUDA compatibility first.

Use `nvidia-smi` in your terminal to confirm the installed CUDA version.

## 📝 Step-by-Step Deployment Guide

### 1.

Setting up the Virtual Environment

**Action:** Create and activate a dedicated Python virtual environment.

**Result:** This isolates project dependencies, preventing conflicts with your system Python installation.

### 2.

Installing Core Dependencies

**Action:** Install PyTorch, Transformers, and the specific SD 3.5 library versions.

**Result:** All necessary foundational libraries are installed and configured for GPU acceleration.

### 3.

Configuring the API Endpoint

**Action:** Create the `config.yaml` file to define the local API v4 endpoints and resource paths.

**Result:** The system knows exactly where to find the model checkpoints and how to communicate with the inference engine.

```yaml
api_version: v4
model_path: /data/sd3.5/checkpoint
api_endpoints:
  generation: http://localhost:7860/generate
  inference: http://localhost:7860/infer
  fine_tune: http://localhost:7860/finetune
```

### 4.

Initial Model Download

**Action:** Download the official SD 3.5 checkpoint weights using the dedicated repository CLI.

**Result:** The base model weights are securely stored and validated for the local environment.

### 5.

Running the Inference Engine

**Action:** Start the local API server using the configured YAML file.

**Result:** The model is loaded into VRAM, and the API is ready to accept generation requests.

### 6.

Generating Test Images (Verification)

**Action:** Submit a simple text-to-image prompt via the API client.

**Result:** A high-quality image is generated quickly, confirming that the entire pipeline is operational.

### 7.

Preparing Fine-Tuning Data

**Action:** Curate your dataset (images + corresponding captions) and organize them into the required directory structure.

**Result:** The data is clean, labeled, and ready for supervised fine-tuning.

### 8.

Running the Fine-Tuning Script

**Action:** Execute the provided fine-tuning script, pointing it to your dataset and the base model.

**Result:** The script iteratively trains the model, optimizing weights based on your specific dataset.

### 9.

Saving the Custom Model

**Action:** Save the newly optimized weights into a separate, versioned directory.

**Result:** You now have a specialized, high-performing checkpoint tailored to your specific aesthetic or subject matter.

### 10.

Testing the Custom Model

**Action:** Restart the inference engine, loading the newly fine-tuned checkpoint instead of the base model.

**Result:** The custom model is fully operational, proving the successful completion of the entire workflow.

## 💡 Expert Tips for Optimization

*   **VRAM Management:** Use gradient accumulation and mixed precision (bfloat16) during fine-tuning.

This significantly reduces VRAM usage without sacrificing stability.

*   **Batch Processing:** When running inference, always utilize batch processing if possible.

Generating multiple images simultaneously drastically improves throughput.

*   **Optimized Sampling:** For the best quality/speed balance, use the DPM++ 2M Karras sampler with a low step count (20-30 steps).

This is often faster than pure DDIM.

## ⚠️ Troubleshooting Common Errors

**Error 1: `CUDA out of memory`**

**Cause:** Your current batch size or resolution is exceeding your GPU's VRAM capacity.

This is common during fine-tuning.

**Fix:** Reduce the batch size by 1 or switch to a lower resolution (e.g., 512x512).

If the issue persists, consider gradient checkpointing.

**Error 2: `ModuleNotFoundError: No module named '...'`**

**Cause:** A required library was not installed in the active virtual environment.

**Fix:** Always run `pip list` inside your virtual environment to verify all dependencies.

Re-run the installation command if necessary.

**Error 3: API endpoint fails with `400 Bad Request`**

**Cause:** The API request payload structure does not match the expected schema defined in your `config.yaml`.

**Fix:** Double-check the documentation for the specific API version (v4).

Ensure all required fields (like `prompt` and `negative_prompt`) are present and correctly formatted (e.g., arrays vs.

strings).
